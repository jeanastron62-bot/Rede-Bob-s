com qualquer fase — foi o que faltou nas Fases 1-4 e custou muitas rodadas):

  1. Cole o conteúdo COMPLETO de cada arquivo criado ou modificado —

CRIAÇÃO DE PEDIDO — uma única função de service, chamada pelas duas rotas,
com um parâmetro `clientOnline: boolean`. Validações, todas server-side:

  1. trailerOpen === false → 403
  2. type === DELIVERY e deliveryActive === false → 403
  3. type === MESA e (tableNumber > maxTables ou < 1) → 400
     (tableNumber é INT no schema — comparação numérica limpa, sem parsing
     de string)
  4. type === DELIVERY sem neighborhoodId válido e active → 400.
     NUNCA existe fallback de taxa.
  5. Para cada item com MenuItem.requiredChoice != null: o selectedChoice
     enviado tem que estar em requiredChoice.options. Ausente ou inválido →
     400 dizendo QUAL item e QUAL escolha faltou.
  6. Item com archived: true ou available: false → 400
  7. paymentMethod === DINHEIRO e cashPaidAmount < total → 400
  8. clientOnline && type === MESA → requiresStaffConfirmation: true.
     Qualquer outro caso → false.
  9. Order + OrderItem[] + OrderItemExtra[] numa ÚNICA prisma.$transaction().

CÁLCULO — a fórmula, exata:

  linha    = (item.unitPrice + Σ(extra.unitPrice × extra.quantity)) × item.quantity
  subtotal = Σ linhas
  total    = subtotal + (type === DELIVERY ? deliveryFeeSnapshot : 0)

  Os extras MULTIPLICAM pela quantidade do item. Se você somar os extras
  uma vez só, "2× X-Burguer com bacon" cobra um bacon — cobrança errada.

  Toda a aritmética com métodos do Prisma.Decimal (.plus(), .times()).
  NUNCA Number(price) * qty — isso reintroduz float justamente onde importa.

  O servidor IGNORA qualquer price/subtotal/total que venha no payload.
  Busca os preços por menuItemId e recomputa do zero.

SNAPSHOTS (obrigatórios):
  MenuItem.name  → OrderItem.menuItemName
  MenuItem.price → OrderItem.unitPrice
  (idem para OrderItemExtra)
  Neighborhood.name       → Order.neighborhoodNameSnapshot
  Neighborhood.deliveryFee→ Order.deliveryFeeSnapshot
  O total SEMPRE usa o snapshot, nunca uma consulta ao valor atual.

PATCH /orders/:id/accept — ATRIBUIÇÃO ATÔMICA:

  const r = await tx.order.updateMany({
    where: { id, type: 'DELIVERY', status: 'PRONTO', assignedToId: null },
    data:  { status: 'EM_ROTA', assignedToId: userId, assignedToName: username },
  });
  if (r.count === 0) throw new ConflictError('Pedido já foi aceito por outro entregador');

  updateMany com where condicional é um compare-and-set atômico. Sem isso,
  dois entregadores veem o mesmo pedido PRONTO, ambos aceitam, ambos dirigem
  ao mesmo endereço. Na mesma transação: OrderStatusHistory + createLog.

MUDANÇA DE STATUS: OrderStatusHistory gravado DENTRO da mesma transação que
atualiza Order.status. As duas escritas juntas ou nenhuma.

  CHAPISTA:   AGUARDANDO→PREPARANDO, PREPARANDO→PRONTO
  ENTREGADOR: EM_ROTA→ENTREGUE (só se assignedToId === ele)
  GARCOM/CHAPISTA/ADM/TI: qualquer→CANCELADO (notes obrigatório)
  ADM/TI: qualquer transição

  Pedido com requiresStaffConfirmation: true NÃO SAI de AGUARDANDO até ser
  confirmado via /confirm.

ESCOPO DO GET /api/orders — POR STATUS, NÃO POR DATA:

  O expediente vai das 18h às 05h — atravessa a meia-noite. Se a cozinha
  filtrasse "pedidos de hoje", à meia-noite todos os pedidos em preparo

useAuthStore → login, token JWT (localStorage), logout. O `logout()` do
               store só limpa token e localStorage — ele NÃO deve importar
               `socket.ts` diretamente (cria import circular). Quem aciona
               o logout na UI (o botão em PanelLayout, Fase 7) chama os
               dois: `useAuthStore.getState().logout()` e
               `useSocketStore.getState().disconnectAll()`. Sem isso, a
               conexão de socket autenticada sobrevive ao logout — em
               troca de turno no mesmo tablet, o próximo usuário herdaria
               a sessão de socket do anterior até um refresh manual.

utils/money.ts     → aritmética em CENTAVOS INTEIROS.
                     Decimal do Prisma chega no JSON como STRING ("29.90"),
                     não number. `toCents` aceita `string | number` e NUNCA
                     multiplica como float — separe a parte inteira da
                     decimal como texto antes de qualquer aritmética:
                       const [w, f = '00'] = String(v).split('.');
                       return parseInt(w, 10) * 100 + parseInt((f + '00').slice(0, 2), 10);
                     Fazer `Number(v) * 100` reintroduz o erro de ponto
                     flutuante que a estratégia de centavos existe pra
                     evitar (ex.: 29.99 * 100 = 2998.9999999999995) — só
                     não quebra por sorte de arredondamento, não por
                     design. Divida por 100 só na formatação final.

                     customerName. Não existe campo `identification` no
                     schema — é derivado.

                 inclui `ingredients: string[]` (Fase 6 precisa pra
                 renderizar a descrição visível do item). O tipo Order
